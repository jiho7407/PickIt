# PickIt — MVP One-Pager

> 사고 싶은데 망설여질 때, 나와 비슷한 상황의 사람에게 현실적인 한마디를 듣고 결정하는 소비 의사결정 서비스.

---

## 1. 문제 정의

20~30대 사용자는 고물가, 고정비, 부채 부담 속에서 소비를 줄이고 싶어 한다. 하지만 실제 구매 결정 순간에는 충동을 멈추게 해주는 장치가 부족하다.

기존 방식의 한계:

1. **가계부** — 이미 쓴 돈을 기록할 뿐, 구매 직전 결정을 돕지 못한다.
2. **SNS 투표** — 가격, 소득, 상황 같은 맥락이 빠진 단순 찬반에 그친다.
3. **지인 상담** — 개인 예산과 소비 습관을 깊이 반영하기 어렵고, 사소한 고민은 묻기 민망하다.
4. **절약 커뮤니티** — 비구조적 대화가 많아 개인의 반복 패턴으로 축적되지 않는다.

결과적으로 사용자는 "줄이고 싶다"는 의지와 "결국 샀다"는 행동 사이에서 반복적으로 후회한다.

---

## 2. 솔루션

사용자가 사고 싶은 상품과 상황을 올리면, 다른 사용자가 "사도 된다 / 사지 말자"로 투표하고 한마디 피드백을 남긴다. 7일 뒤에는 실제 구매 여부와 만족/후회를 기록하게 해, 한 번의 고민을 소비 습관 데이터로 전환한다.

핵심 차별점:

- **결정 전 개입**: 소비 이후 반성이 아니라 구매 직전 판단을 돕는다.
- **맥락 있는 피드백**: 가격, 상황, 사용자 조건을 함께 보여준다.
- **행동 학습**: 7일 후 실제 선택과 감정을 기록한다.
- **절약 보상**: 안 산 금액을 절약 성과로 보여준다.

---

## 3. 핵심 가설

| # | 가설 | 검증 방법 |
| --- | --- | --- |
| H1 | 사용자는 구매 직전 타인의 현실적인 제동을 원한다 | 고민 작성 전환율, 고민당 투표 수 |
| H2 | 투표 결과와 한마디는 구매 확신 또는 절제에 영향을 준다 | 결과 확인율, 7일 후 구매/비구매 기록 |
| H3 | 절약 금액을 보여주면 재방문 동기가 생긴다 | 회고 응답률, 절약 기록 재방문율 |

---

## 4. 핵심 사용자

| 사용자 | 설명 | 주요 니즈 |
| --- | --- | --- |
| 고민 작성자 | 사고 싶은 물건이 있지만 확신이 없는 사람 | 충동 제어, 현실적인 피드백, 결과 확인 |
| 투표 참여자 | 다른 사람의 소비 고민에 의견을 주는 사람 | 가벼운 참여, 공감, 비슷한 상황의 고민 보기 |
| 운영자 | 서비스 품질과 리텐션을 관리하는 사람 | 방치 고민, 회고 대상, 알림 후보 확인 |

---

## 5. 핵심 상태 모델

```text
Draft → Open → Decided → FollowupDue → FollowedUp → Archived
```

| 상태 | 의미 |
| --- | --- |
| Draft | 작성 중, 아직 공개되지 않음 |
| Open | 공개되어 투표/의견을 받는 중 |
| Decided | 작성자가 결과를 확인하고 구매 결정을 내린 상태 |
| FollowupDue | 생성 후 7일이 지나 회고 입력이 필요한 상태 |
| FollowedUp | 실제 구매 여부와 만족/후회를 기록한 상태 |
| Archived | 오래된 고민 또는 운영상 숨김 처리된 상태 |

## 6. 데모 장면

```text
[홈/목록]
  사용자가 다른 사람의 소비 고민을 둘러본다.
                    ↓
[고민 작성]
  상품명, 가격, 이미지, 상황을 입력해 고민을 올린다.
                    ↓
[투표]
  다른 사용자가 "사도 된다 / 사지 말자"에 투표하고 한마디를 남긴다.
                    ↓
[결과]
  작성자는 투표 비율과 의견을 확인한다.
                    ↓
[7일 후 회고]
  작성자는 실제로 샀는지 참았는지 기록한다.
                    ↓
[절약 기록]
  참은 경우 상품 가격만큼 절약 금액이 누적된다.
```

## 7. MVP IN / OUT

### IN

- 공개 고민 목록
- 고민 작성: 이미지, 상품명, 가격, 카테고리, 상황
- 고민 상세
- 투표: 사도 된다 / 사지 말자
- 짧은 한마디 의견
- 결과 비율 표시
- 7일 후 회고 입력
- 절약 금액 계산
- 최소 인증: 고민 작성자 식별, 회고 권한 보호
- Supabase RLS 기반 데이터 보호

### OUT

- 카카오톡 실제 발송
- 복잡한 추천/매칭 알고리즘
- 앱 푸시
- 월간 소비 분석 리포트
- 결제/커머스 연동
- 관리자 대시보드 전체 구현
- 네이티브 앱

## 10. 완료 기준

MVP가 끝났다고 말하려면:

1. 사용자가 고민을 작성하고 공개할 수 있다.
2. 다른 사용자가 로그인 없이 또는 익명 세션으로 투표할 수 있다.
3. 같은 사용자는 같은 고민에 중복 투표할 수 없다.
4. 작성자는 결과 비율과 의견을 확인할 수 있다.
5. 생성 후 7일이 지난 고민은 회고 대상이 된다.
6. 작성자는 실제 구매 여부와 만족/후회를 기록할 수 있다.
7. 구매하지 않은 경우 절약 금액이 누적된다.
8. 위 흐름이 Playwright E2E로 검증된다.

## 11. UI IA / Screen Map

Figma의 임시 그룹/노드 id는 Planning의 고정 레퍼런스로 남기지 않는다. 디자인은 수정될 수 있고, 개발 시점에 필요한 화면 프레임을 다시 선택해 전달받아 구현한다.

현재 확인된 큰 화면 범주:

```text
스플래시 화면
온보딩
홈 = 투표
투표 상세
  - 살지/말지 투표
  - A/B 투표
나의투표, 댓글달기, 투표 및 댓글 삭제
투표 만들기
  - 살지/말지 투표
  - A/B 투표
프로필
  - 기본 프로필
  - 프로필 내 목록
  - 소비기록
    - 소비리포트 이동
    - 소비여부 물어보기 (프로필 안에서)
    - 소비여부 물어보기 (소비리포트 안에서)
  - 알람
  - 에러화면
```

개발 시 Figma 사용 원칙:

- 섹션 단위 묶음은 플로우 파악용으로 사용한다.
- 개별 폰 화면 프레임은 실제 UI 구현 스펙 추출용으로 사용한다.
- 임시 그룹 node-id는 문서에 영구 참조로 남기지 않는다.
- 개발 직전에 해당 태스크에 필요한 화면만 다시 읽는다.
